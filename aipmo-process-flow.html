❌ Verification Failure Flow (Section 4.6)
🟡 MINOR FAIL
Feature works but edge case missing → 15-min Fix Window → Re-verify
🔴 MAJOR FAIL
Feature broken → 1 Fix Sprint (60 min) → Re-verify
Second failure: EVENT: VERIFICATION_ESCALATED → Posted to #akira-swarm → Human PM review.
Max 1 retry per failed item per release cycle. Fix sprints consume sprint capacity.
🔗 Cross-Squad Dependency Resolution (Section 4.7)
Level 1 — Pre-Sprint Planning: Program Manager sequences work so dependent items are staggered across sprints.

Level 2 — Runtime Dependencies: DEPENDENCY_BLOCKED event → 15-min resolution timeout → RESEQUENCE/MOCK/SWAP/ESCALATE
Key principle: Dependencies should be caught in pre-sprint planning. Runtime dependencies indicate a planning gap — logged for retrospective improvement.

🛡️ Anti-Fabrication Protocol (Section 10)
Core Principles:
1. Strict Evidence: Claims based only on execution context (code output, test results, logs)
2. Source Citation: Every status must reference Git SHA, deployment URL, or test output
3. Missing Info: "Evidence not available for [item]. Cannot verify." Never fabricate.
4. No Inference: "Code compiles" ≠ "feature works". Each AC requires specific evidence.
5. Verification Traceability: Every PASS/FAIL links to artifact. No pass without evidence.
Checkpoint Gate Check: Does every "COMPLETE" claim have linked evidence? VERIFICATION_SUSPECT flags require FC review.
📅 Example Timeline (4 Sprints = 1 Release)
Pre-Sprint (Before 9:00 AM):
• AI Estimator: Backlog decomposed → ESTIMATES_COMPLETE
• AI Program Manager: Work assigned to squads → BACKLOG_READY
• AI Portfolio Manager: Budget validated → BUDGET_APPROVED
• Human PM: Scope approved → HUMAN_APPROVED
• Requirements Verifier + All Squads: Contracts signed → BEGIN_WORK
Sprint 1 (9:00 AM - 10:30 AM):
9:00 AM - BEGIN_WORK → All 3 squads start simultaneously
Squad Alpha (Backend)
Squad Beta (Frontend)
Squad Gamma (QA)
9:30 AM - Reality Check — each FC checks their squad
10:00 AM - Sprint execution ends → TASK_COMPLETE
10:00-10:30 AM - Verification (~30 min overhead)
10:30 AM - Checkpoint Gate #1 published
Sprint 2 (10:30 AM - 12:00 PM):
Similar pattern: BEGIN_WORK → Reality Check @ 11:00 → Complete @ 11:30 → Gate @ 12:00
Sprint 3 (12:00 PM - 1:30 PM):
Similar pattern: BEGIN_WORK → Reality Check @ 12:30 → Complete @ 1:00 → Gate @ 1:30
Sprint 4 (1:30 PM - 3:00 PM):
Similar pattern: BEGIN_WORK → Reality Check @ 2:00 → Complete @ 2:30 → Gate @ 3:00
Post-Release (3:00 PM - 3:30 PM):
3:00-3:15 PM - Strategic Alignment Review
3:15-3:30 PM - Retrospective (automated learning integration)
3:30 PM - Human Review Checkpoint (PM reviews when available)
Result: 4 sprints × 3 parallel squads = 12 sprint-equivalents in ~6.5 hours
